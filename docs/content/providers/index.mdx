A provider is a small package that implements the infra-ts [entity contract](/standard). Most are a thin wrapper over the provider's REST API. A few (like `stripe-projects`) instead **compose a vendor CLI**, so you can declare infra that another tool provisions. Either way, entities persist only identity state, stay idempotent, and emit typed env.

The umbrella `infra-ts` package bundles the first set as subpath imports:

```ts
import { NeonProject } from "infra-ts/neon";
import { VercelProject } from "infra-ts/vercel";
import { UpstashRedis } from "infra-ts/upstash";
```

## How providers work

Every entity implements the same small lifecycle:

- `read` fetches live remote state
- `diff` compares your desired config to that state
- `provision` creates or updates the resource (idempotent)
- `pullEnv` returns the typed env it exposes
- `deprovision` tears it down

The engine handles graph ordering, refs, state I/O, env output, hooks, and CLI rendering. See [The standard](/standard) for the full contract and [Core concepts](/concepts) for entities, refs, and identity state.

### REST vs CLI (transport is the entity's choice)

For **reconciled config** (projects, env vars, domains) providers stay thin: `typed options → REST read → pure diff → REST mutation`. For **imperative, vendor-owned actions** — building, deploying, or provisioning through another tool — an entity can instead compose the vendor CLI via an `exec` capability, declaring the CLIs it needs with `requiredTools()` (the engine detects and installs them, preferring ephemeral `npx`). `VercelDeployment` uses this to delegate to the `vercel` CLI; the [`stripe-projects`](/providers/stripe-projects) provider uses it to provision through `stripe projects`. Either way an entity still persists only identity state and emits typed env — so REST vs CLI is a free, per-entity choice.

## Credentials

Each provider resolves credentials from the environment (or a provider CLI's login cache) — no secrets are ever written to infra-ts state. The credential env var is listed on each provider's page (e.g. `NEON_API_KEY`, `VERCEL_TOKEN`). Set them in your shell/CI, or run `infra login` for providers that support CLI OAuth (Neon, Vercel).
