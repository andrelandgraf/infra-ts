<Steps>
  <Step title="Install or run the CLI">
    Install globally to get the `infra` command.

    ```bash
    npm i -g infra-ts
    infra --help
    # or run ad hoc, without installing
    npx infra-ts --help
    ```

    `infra` and `infra-ts` are the same CLI — use whichever you prefer.

  </Step>
  <Step title="Create infra.ts">
    Declare your resources as typed entity instances. Scope entities hold the org/team you provision into, and resources point at them (and each other) through refs — so just wiring things together builds the dependency graph for you.

    ```ts
    import { defineInfra } from "infra-ts";
    import { NeonOrg, NeonPostgres, NeonProject } from "infra-ts/neon";
    import { VercelProject, VercelTeam } from "infra-ts/vercel";

    const neon = new NeonOrg({ name: "neon" });
    const vercel = new VercelTeam({ name: "vercel" });
    const project = new NeonProject({ name: "web-neon", region: "aws-us-east-1", org: neon.id });
    const db = new NeonPostgres({ name: "db", projectId: project.id });

    export default defineInfra({
      entities: [
        neon,
        vercel,
        project,
        db,
        new VercelProject({
          name: "web-vercel",
          team: vercel.id,
          env: { DATABASE_URL: db.env.databaseUrl },
        }),
      ],
    });
    ```

  </Step>
  <Step title="Authenticate and link scopes">
    `login` runs each provider's OAuth flow through the CLIs you already use (`neonctl`, `vercel`). `link` then lets you pick the org/team for each scope entity and records it in `.infra/<env>.json`.

    ```bash
    infra login   # OAuth via the provider CLIs
    infra link    # pick an org/team per scope → .infra/<env>.json
    ```

    Prefer explicit credentials in CI? Set them in the environment instead:

    ```bash
    export NEON_API_KEY=...
    export VERCEL_TOKEN=...
    ```

  </Step>
  <Step title="Preview and apply">
    `plan` reads live remote state and shows changes without mutating anything.

    ```bash
    infra plan
    infra apply
    infra status
    ```

  </Step>
</Steps>

:::tip
The first `apply` writes `.infra/<env>.json`, a tiny identity link file. Keep it local and uncommitted, just like `.env.local` or `.vercel`.
:::

## Pull typed environment values

Entities can expose typed environment outputs. `checkout` pulls those values from live resources and writes `.env.<env>`.

```bash
infra checkout --env local
```

In application code, `parseEnv` validates the process environment without network access:

```ts
import infra from "./infra";
import { parseEnv } from "infra-ts";

const env = parseEnv(infra);
env["db"].databaseUrl;
```

## Run with resolved env

`infra run` checks out env values in memory and runs a child command with those variables injected.

```bash
infra run -- npm run dev
```
