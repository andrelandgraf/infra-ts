infra-ts lets you describe your whole stack as typed entities in one `infra.ts` file, then keeps it in sync with the real thing by talking to each provider's API.

There's no heavy state backend to babysit. infra-ts just remembers which remote resource maps to which entity, re-reads the live state on every command, and treats the provider APIs as the source of truth.

## Why it exists

Terraform and Pulumi are great — but they're pretty heavy for vibe coding a full-stack TypeScript app. They make a lot of sense for sophisticated, enterprise cloud management. They make less sense when you (or your agents) just want to assemble Vercel, Neon, and Upstash into a full-stack app and move on.

infra-ts is the lightweight take for that world:

- **Identity state only.** `.infra/<env>.json` just maps entity names to provider ids — no resource attributes, no secrets, no snapshot to babysit.
- **Live reconciliation.** `plan`, `apply`, `status`, and `checkout` read the current remote state over REST before deciding what changed.
- **TypeScript-native.** Providers are typed classes with refs and env outputs your editor and CI already understand.
- **Open providers.** A provider is just a thin package over a REST API — there's no custom engine to learn.

## A minimal `infra.ts`

```ts
import { defineInfra } from "infra-ts";
import { NeonOrg, NeonPostgres, NeonProject } from "infra-ts/neon";
import { VercelProject, VercelTeam } from "infra-ts/vercel";

const neon = new NeonOrg({ name: "neon" });
const vercel = new VercelTeam({ name: "vercel" });
const project = new NeonProject({
	name: "my-app-neon",
	region: "aws-us-east-1",
	org: neon.id,
});
const db = new NeonPostgres({ name: "my-db", projectId: project.id });

export default defineInfra({
	entities: [
		neon,
		vercel,
		project,
		db,
		new VercelProject({
			name: "my-app-vercel",
			team: vercel.id,
			framework: "nextjs",
			env: {
				DATABASE_URL: db.env.databaseUrl,
			},
		}),
	],
});
```

A scope entity is just the org/team you provision into: `infra login` signs you in and `infra link` picks the scope. See [Core concepts](/concepts#orgsteams).

```bash
infra plan
infra apply
infra status
```

## What ships today

The `infra-ts` npm package is the batteries-included CLI and SDK — the core runtime plus the first set of providers, all as subpath imports:

```ts
import { defineInfra, parseEnv } from "infra-ts";
import { NeonProject } from "infra-ts/neon";
import { VercelProject } from "infra-ts/vercel";
import { StripeProduct } from "infra-ts/stripe";
// or let Stripe Projects provision across providers:
import { NeonPostgres } from "infra-ts/stripe-projects";
```

It's early days, but the idea is deliberately simple: typed entities, live REST reconciliation, no attribute state.
