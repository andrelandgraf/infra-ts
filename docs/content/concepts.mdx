## Entities

An entity is one provisionable resource. A Neon project, Postgres database, Vercel project, Stripe product, or OpenAI key can each be modeled as a class instance.

Entities expose:

- a stable `name`, chosen by you
- typed constructor options
- typed output refs such as `entity.id` and `entity.env.databaseUrl`
- lifecycle methods for `read`, `diff`, `provision`, `pullEnv`, and `deprovision`

## Refs

Refs are deferred values from another entity. Passing a ref into a constructor does two things:

1. creates a dependency edge in the graph
2. resolves to the live value when `apply` runs

```ts
const vercel = new VercelTeam({ name: "vercel" });
const db = new NeonPostgres({ name: "db", projectId: project.id });

new VercelProject({
	name: "web",
	team: vercel.id,
	env: {
		DATABASE_URL: db.env.databaseUrl,
	},
});
```

## Orgs/teams

Before infra-ts can provision anything, it needs two per-developer things you don't want hardcoded in `infra.ts`: a token, and the org/team to provision into. Both are modeled by a provider-native scope entity such as `NeonOrg` or `VercelTeam` — a named entity that creates no remote resource of its own.

```ts
import { NeonOrg, NeonProject } from "infra-ts/neon";

const neon = new NeonOrg({ name: "neon" });
const project = new NeonProject({
	name: "web",
	region: "aws-us-east-1",
	org: neon.id,
});
```

- `infra login` authenticates the scope entity's provider (OAuth passthrough via `neonctl` / `vercel`).
- `infra link` lists your orgs/teams, you pick one, and the scope id is written to `.infra/<env>.json`.
- Entities reference the scope with `org: neon.id` (Neon) or `team: vercel.id` (Vercel). The scope entity resolves first, so the scope is ready before anything provisions.

Two scopes of the same provider just get two names (`new NeonOrg({ name: "work" })`). The common case is one login with many orgs (shared token, scope differs per scope).

When a provider's cached CLI token is a short-lived OAuth token (e.g. Neon's), infra-ts self-heals: on a `401` it refreshes the token via the CLI and retries once, so an idle session doesn't fail your command.

## Identity state

infra-ts does not store resource attributes. It stores only the binding needed to remember which remote resource belongs to which entity.

```json
{
	"version": 1,
	"environment": "local",
	"entities": {
		"db": { "id": "br-wandering-frost-12345678" }
	}
}
```

That file is deliberately small:

- no resource attributes
- no secrets
- no drift-prone snapshot
- no lock server

## Live reconciliation

Every command re-reads provider APIs:

```txt
infra.ts declaration
        +
.infra/<env>.json identity bindings
        +
live provider API reads
        =
plan / apply / status / checkout
```

If the remote drifted, infra-ts sees that drift from the provider API directly. If a resource was renamed or recreated, the identity binding tells infra-ts which resource to compare.

## Composition & provider-owned state

infra-ts is a composition layer over the tools you already use — it doesn't try to own everything. A provider can wrap a REST API directly, or **compose a vendor CLI** (like `stripe-projects` driving `stripe projects …`). Either way, the same rules hold:

- **Compose, don't capture.** The vendor CLI stays fully usable; infra-ts owns only the composition graph in `infra.ts`. Delete `infra.ts` and your native tools still work.
- **Providers own where their state lives.** A provider's `read`/`provision` decide whether identity comes from a native file, a live CLI read, or infra-ts's own state file. When identity can be resolved live (e.g. by the entity's `name`), the provider persists nothing at all — and even then, infra-ts state only ever holds ids and content hashes, never secrets or attributes.

## Environments

Every command targets an environment:

```bash
infra plan --env preview
infra apply --env production
```

The environment selects:

- `.infra/<env>.json` identity bindings
- `.env.<env>` output file
- environment-specific credentials or defaults

infra-ts does not use `NODE_ENV` for this selection.
